## 2. Decision Cycles

1. Cycle ID: A1-D1
   - Proposal (driver): Add `consecutive_errors: u32` counter inside `http_poll_stream`
     loop; yield `ListenerError::ReconnectExhausted` and return when the counter reaches
     `max_reconnect`. Reset counter to 0 on any success (block_number or get_logs).
   - Critique (navigator): Agreed. `consecutive` is the critical qualifier ŌĆö isolated
     transient failures should not accumulate toward termination. Reset on block_number
     success (not only get_logs success) so a recovered RPC clears the state immediately.
   - Resolution decision: Implemented as proposed. Counter resets on `block_number` Ok
     and on `get_logs` Ok. `subscribe()` now passes `max_reconnect` to `http_poll_stream`.
   - Decision owner: Rust Backend Specialist
   - Linked code/test evidence: `src/eth_node/src/events.rs` ŌĆö `http_poll_stream`
   - Follow-up action: Update `max_reconnect` field doc and `ReconnectExhausted` display
     to remove "WebSocket"-only framing.

2. Cycle ID: G3-D1
   - Proposal (driver): Use an inline tokio TCP proxy (no external tool). Bind `TcpListener`

   - Critique (navigator): Preferred over toxiproxy ŌĆö eliminates external dependency,
     works reliably on Windows, and is self-contained in the test file. One connection
     at a time is sufficient for the reconnect test.
   - Resolution decision: Inline proxy implemented in `spawn_tcp_proxy`. No changes
     to capture-session.sh needed.
   - Decision owner: GitHub Copilot (navigator)
   - Linked code/test evidence: `src/eth_node/tests/integration_events.rs` ŌĆö
     `spawn_tcp_proxy`, `test_ws_reconnect_exhausted_after_proxy_abort`
   - Follow-up action: None.
